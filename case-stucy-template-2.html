PROJECT NAME

Overview

The project began with the objective to create a CSS style sheet and 10-15 HTML templates for teachers to create new courses with and students to access course material through. The templates will be used by teachers to create courses through the Brightspace learning platform. My job was to design and develop templates for teachers to input content into.

What I learned

INSIGHTS:

CHALLENGE

Tedious copying and pasting during summer vacation
Lack of instruction on how to operate new template. Frustration about wasted time.

SOLUTION

Fear of being misrepresented by sub par product.
Pressure to learning new system before delegating to teachers.

GOING FORWARD

Disinterest in course material
Fear of missing an assignment or quiz
Overwhelmed with amount of information.

The Initial Design

Initially, I provided a CSS sheet along with 12 html templates in which teachers could copy and paste course content into. I provided this, along with an instruction manual, co-written with leadership within AESA, which instructed teachers on migrating courses from their previous online learning platform.

BUT, in testing . . .

Before roll out to teachers, I was able to recruit a few teachers to test the templates before 'go-day'. I was able to see the courses they were building in real time, as well as receive feedback on the usefulness and usability of each template and element.

Some curious patterns arose:
  • I provided 12 templates, but ~90% of were made using 1 specific template! Huh...
  • Participants were copying and pasting elements from one template to another. This produced some clever designs. When unregulated, it also produced some unexpected usability, accessibility, responsively and styling errors.
  • Participants' content was more varied and engaging than I had anticipated.
  • In transferring content, participants often accidentally deleted template elements, brought in accidental formatting from old learning platform.

Pain Points (of teachers)

  • Addientally deleting parent elements of the child elements that they are trying to edit
  • Tedious copying and pasting from 'migrated' course content into templates
  • Inability to change page 'type' mid way through page creation
  • Confused with needing multiple tabs and programs open in order to create content
  • Feeling stuck when assets don't load correctly or get lost in migration between learning platforms
  • Tedious restructuring of content because of standardization of template usage amongst courses
  • Unable to reuse some elements throughout page.
  • Accidentally creating usability issues with unexpected components usage and forgetting alt text on images.

Design Solutions

  • I created a more advanced 'elements page' for teachers to be able copy and pasted the most commonly deleted content.
  • Worked with teacher-testers and administrators to formalized content migration process. Edited HTML/CSS to increase usability on select templates while offering more advanced components for teachers to use as needed.
  • Increase design autonomy of teachers by adding icon components to elements page.
  • Taught teachers about repositories so that they could check and change common errors on repository paths themselves!
  • Worked with admins to gradually but clearly identify guidelines and documentation to aid uniform course creation amongst many classes.
  • Lead accessibility workshop for teachers and corrected HTML issues (just needed to change some IDs to classes, to that teachers can resume elements without error) in order to work with the way teachers preferred to use templates.

In Summary

Throughout the course of implementation of this product many more surprises and opportunity for improvements arose. I love talking about this project for many reasons. From a UX perspective, this project really started to shine a light for me on the value of research and testing throughout the course of a product creation. There were so many valuable and unexpected insights that came throughout the course of implementation, which I now heavily value seeking out earlier in the design process!

While I am proud of the product that I was able to deliver (along with the creative genius of some truly remarkable teachers and administrators), had I known what I know now, here are some things I may have done differently;

What I learned:
  • START WITH MORE RESEARCH!
    Starting off on the project, I thought my job was to provide a CSS and HTML templates for teachers to use to create courses for their online learning platform. Had I more deeply explored the way that teachers were using their previous platform earlier in the design process, I would have earlier identified the need to deliver a template/components usage guidelines (along with documentation for it) in order to ensure easier implementation of it.
    ‍
  • TEST EARLIER AND OFTEN!
    While we were limited by the fact that the teachers were on summer vacation, once teachers were brought into the process, many were more than willing to provide feedback when requested. I could have been taking advantage of this much earlier on in the process!
    ‍
  • NEW THINGS SHOULD COME WITH DOCUMENTATION (and guidelines)!
    While it was helpful for teachers to have new and beautiful components and templates to build out their classes, it wasn't much help without knowing how to use them. After roll out, we quickly realized that reachers were starting to use components to build courses that varried more extensively from the desired design that expected. Thus came the need for guidelines. We were then in a position to rushedly determine guidelines and write documentation for the new features so that teachers could create uniform courses across departments. Had we been planning this before roll out, we would have had more time to collect feedback and been more intentional on the guidelines and decisions being made.
    ‍
  • EVERY STEP HAS A USER:
    It is not enough to design for the end user. There are many users in between, whose experience matters just as much to the end result. While I was asking a lot of the right questions earlier on, I was not asking them to all right people! Making sure my user interviews represent the target population would have afforded me less backtracking later on in the project.
